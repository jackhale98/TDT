Team Collaboration

Simple workflow commands for teams who want structured reviews without being git experts

🤝 Collaboration Made Simple

TDT includes optional workflow commands that let your team review and approve design changes using simple commands. You don't need to be a git expert—TDT handles the complexity behind the scenes.

This feature is completely opt-in. If you prefer managing git yourself, these commands won't get in your way. But if you want a guided experience with built-in review processes, read on.

💡 What This Enables

With workflow enabled, you can submit requirements for review, get approval from qualified teammates, and release changes—all with simple tdt commands that create proper GitHub/GitLab pull requests automatically.

📖 Key Concepts

Before diving in, here are a few concepts that will help:

📝

Status

Every entity (requirement, risk, test, etc.) has a status that tracks where it is in the review process: Draft, Review, Approved, or Released.

👥

Team Roster

A simple file that lists who's on your team, their roles, and what they can approve. This is how TDT knows who has permission to sign off on changes.

📥

Submit

When you're ready for review, submit your work. TDT creates a branch, updates the status, and opens a pull request for you.

✅

Approve

Reviewers check the changes and approve. TDT records who approved, updates the status, and can merge the pull request automatically.

🔄 The Review Workflow

Here's how changes flow through the system:

✏️

👀

✅

💡 If changes are rejected during review, they go back to Draft for revision.

What happens at each step

Status What it means Who can transition
Draft Work in progress. Edit freely until ready for review. Author submits for review
Review Submitted for review. A pull request is open and awaiting approval. Qualified reviewer approves or rejects
Approved Reviewed and approved. Ready to be released. Management releases to production
Released Official version. This is the controlled document. Final state (can be made obsolete)

⚡ Quick Start

Get your team set up in just a few minutes:

Enable Workflow in Config

Add these lines to your .tdt/config.yaml file:

workflow:
  enabled: true
  provider: github    # or: gitlab, none
  base_branch: main   # your default branch

Create Your Team Roster

Initialize a team roster file that defines who can approve what:

tdt team init

Then add your team members:

tdt team add --name "Jane Smith" --email jane@example.com \
    --username jsmith --roles engineering,quality

Submit Work for Review

When you're ready, submit your changes. TDT creates a branch and pull request:

# Submit a specific requirement
tdt submit REQ@1

# Submit all draft requirements
tdt submit --type req --all

Review and Approve

Reviewers check the pull request, then approve:

# See what's waiting for your review
tdt review list

# Approve an item
tdt approve REQ@1

🎭 Team Roles

TDT uses a simple role system to control who can approve what:

Engineering

Engineering

Can approve requirements, components, features, and other technical entities.

Quality

Quality

Can approve tests, NCRs, CAPAs, risks, and quality-related documentation.

Management

Management

Can approve risks and release approved entities to production status.

Admin

Admin

Can approve any entity type. Use sparingly for team leads or project managers.

Team members can have multiple roles. For example, a senior engineer might have both Engineering and Quality roles.

📋 Example Team File

Your .tdt/team.yaml might look like this:

version: 1
members:
  - name: "Jane Smith"
    email: "jane@example.com"
    username: "jsmith"
    roles: [engineering, quality]
    active: true

  - name: "Bob Wilson"
    email: "bob@example.com"
    username: "bwilson"
    roles: [quality, management]
    active: true

approval_matrix:
  REQ: [engineering, quality]
  RISK: [quality, management]
  TEST: [quality]
  _release: [management]

💻 Command Reference

Here are all the workflow commands you'll use:

📤 Submit for Review
# Submit specific items
tdt submit REQ@1 REQ@2

# Submit all draft items of a type
tdt submit --type req --all

# Submit with a message
tdt submit REQ@1 -m "Initial pump requirement"

# See what would happen without doing it
tdt submit REQ@1 --dry-run
✅ Approve Items
# Approve specific items
tdt approve REQ@1

# Approve with a comment
tdt approve REQ@1 -m "Looks good, values verified"

# Approve and merge the pull request
tdt approve REQ@1 --merge
❌ Reject Items
# Reject with a reason (required)
tdt reject REQ@1 -r "Values need to be verified against spec"
🚀 Release Approved Items
# Release specific items
tdt release REQ@1

# Release all approved requirements
tdt release --type req
👀 Review Queue
# See what's waiting for your review
tdt review list

# See all pending reviews (not just yours)
tdt review list --all

# Get a summary of workflow status
tdt review summary
👥 Team Management
# Initialize team roster
tdt team init

# List team members
tdt team list

# See your own role
tdt team whoami

# Add a team member
tdt team add --name "Jane Smith" --email jane@co.com \
    --username jsmith --roles engineering,quality

# Remove a team member
tdt team remove jsmith

🔗 GitHub & GitLab Integration

TDT integrates with GitHub and GitLab to create real pull/merge requests. This means your review workflow shows up in your normal code review tools.

Setting Up GitHub

First, install and authenticate the GitHub CLI:

# Install GitHub CLI (macOS)
brew install gh

# Install GitHub CLI (Windows)
winget install GitHub.cli

# Login to GitHub
gh auth login

Then configure TDT to use GitHub:

# In .tdt/config.yaml
workflow:
  enabled: true
  provider: github
  base_branch: main

Setting Up GitLab

For GitLab, use the GitLab CLI:

# Install GitLab CLI
brew install glab

# Login to GitLab
glab auth login
# In .tdt/config.yaml
workflow:
  enabled: true
  provider: gitlab
  base_branch: main

No Provider (Manual Mode)

If you're using a different git host, or prefer to create pull requests manually, set provider to none: 

workflow:
  enabled: true
  provider: none    # Creates branches but no PRs

TDT will still create branches and commits, but you'll create pull requests yourself.

💡 Tips for Success

✅ Use Dry Run First

Not sure what a command will do? Add --dry-run to see exactly what git commands would run without actually doing anything.

✅ Verbose Mode Shows Details

Add -v or --verbose to any command to see the git commands as they execute.

⚠️ Username Must Match Git

Your username in the team roster must match your git config user.name. Run tdt team whoami to verify you're recognized.

💡 Batch Reviews Are All-or-Nothing

When you submit multiple items together, they create one pull request. The reviewer must approve or reject all of them together.

❓ Common Questions

Do I have to use these workflow commands?

No! The workflow is completely opt-in. If workflow.enabled is false (the default), these commands won't even appear in help. You can always manage git manually if you prefer.

What if I don't have a team roster?

Without a team roster, there are no permission checks. Anyone can approve anything. This is fine for solo projects or small teams without formal processes.

Can I still use git directly?

Absolutely. These commands are just convenient wrappers. You can always use git, gh, or glab directly. TDT stores everything in plain YAML files, so there's no lock-in.

What if the reviewer is on vacation?

If someone with the right role isn't available, use --force to bypass authorization checks. This is logged in the entity file for traceability.

How do I see the git commands TDT runs?

Use --verbose on any command to see each git command as it runs, or use --dry-run to preview without executing.
